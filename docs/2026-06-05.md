# 2026-06-05 — Chia sẻ màn hình (screen share) qua Daily.co — Phase 1

## Mục tiêu phiên

App đã có canvas chung, chat, phiên dịch, gọi audio — **còn thiếu chia sẻ màn hình** (1 người Present → mọi người xem), tính năng cơ bản của mọi nền tảng họp. Phiên này **bàn plan kỹ → chọn hạ tầng → code Phase 1**.

## Quyết định hạ tầng (bàn kỹ trước khi code)

Khảo sát toàn cảnh build-vs-buy cho media (SFU vs CPaaS):

- **Self-host SFU (LiveKit/mediasoup/Janus) → LOẠI:** chủ dự án là vibe coder, không team IT → không nuôi server media.
- **Cloudflare Realtime → LOẠI cho giờ:** rẻ + hợp hạ tầng CF, nhưng **không có recording sẵn** → phải tự xây "recorder bot" + 1 server 24/7 (vài tuần kỹ sư) → quá sức.
- **Zoom Video SDK → LOẠI:** mạnh + phủ TQ, nhưng web SDK đòi **SharedArrayBuffer / cross-origin isolation** → vỡ phần load R2/font/IFC cross-origin của app. Tích hợp khó nhất.
- **Agora → không cần:** mạnh nhất cho Trung Quốc, nhưng phiên chốt "quên China đi".
- **→ CHỌN Daily.co (managed):** WebRTC thuần (không vướng cross-origin isolation), recording sẵn, DX sạch, **zero-ops** (không nuôi server — chỉ thêm 1 hàm token vào Worker). Giá ~$0.0015–0.004/người/phút, free 10k phút/tháng.

Kiến trúc chốt: **Daily = ống dẫn media; socket Collab sẵn có = presence + single-share lock; lazy-join** (chỉ nối Daily khi có người share → tiết kiệm phút). Token mint server-side, `DAILY_API_KEY` chỉ Worker thấy (ở `worker/.dev.vars`, đã gitignore).

## REVIEW KIẾN TRÚC PRODUCTION + QUYẾT ĐỊNH (chốt 2026-06-05, chiều)

Soi "production có gì chưa tối ưu" → chốt hướng. Điểm yếu lớn nhất khớp tham vọng enterprise: **(1) Worker API không auth** và **(2) audio mesh không scale**.

### Kiến trúc hiện tại (3 kênh tách biệt + storage)
- **① socket.io :3002** — sync canvas/cursor + chat/reactions/raise-hand + **presence screen-share & lock**; cũng là signaling cho audio mesh. E2E mã hoá.
- **② Audio mesh WebRTC P2P** — tiếng nói, signaling ghép qua ①, TURN/STUN Cloudflare; STT bám mic local. **Không scale > ~6-8 người.**
- **③ Daily.co SFU** — screen share (Phase 1); token từ Worker; presence/lock qua ①.
- **④ Worker :8787** — R2 (scene/file/chat/library blob mã hoá) + D1 (projects/meetings) + mint Daily token. **CORS `*`, KHÔNG auth (test phase).**

### Quyết định
1. **Auth → Supabase Authentication** (thay Cloudflare Access SSO trong June plan — nhẹ đầu cho solo dev). Worker verify **Supabase JWT** ở middleware → gate mọi route + token endpoint. **Giải quyết:** Worker no-auth, token endpoint mở, identity thật. **KHÔNG giải quyết:** room-key-lưu-server (E2E — vấn đề mã hoá), scaling, rate-limit; per-meeting authz chỉ một phần (Supabase cho danh tính, vẫn phải tự viết luật membership).
2. **Audio → dời lên Daily SFU (CHỐT LÀM NGAY).** Bỏ mesh P2P → audio chạy SFU như screen share: scale N người, gộp 1 stack media, mở khoá recording hợp nhất. Rủi ro đã map (đụng STT/recorder/speaking-detect/listen-only) → làm cẩn thận từng bước, giữ nguyên các atom UI (`audioStateAtom`/`audioRoomInstanceAtom` → ParticipantsBar/CallControls không phải sửa).
3. **Room server → Durable Objects** (June plan): thay socket.io :3002 đơn lẻ → scale + HA.

### Làm SAU (nợ kỹ thuật chấp nhận giai đoạn này)
- Rate limiting / chống abuse Worker.
- Recording persist + R2 lifecycle/cleanup (orphan bytes) + dọn Daily room.
- Observability server-side (room + worker).
- Data residency / compliance (R2 region, Daily region) cho client xuyên quốc gia.
- Room key: hardening E2E (đừng lưu server) — vẫn để mở.

### Roadmap cập nhật (thay Phase 2/3 cũ ở trên)
1. **Phase 2 — Audio → Daily SFU** ⭐ tiếp theo: `DailyAudio` thay `AudioRoom` (mirror `DailyScreenShare`); re-bind STT (lấy mic stream từ Daily) + recorder; giữ atoms để UI không đổi. Test 2-tab kỹ từng bước.
2. **Phase 3 — Recording → R2**: giờ audio+screen cùng Daily → Daily cloud recording (webhook → copy về R2, vì Daily ghi thẳng R2 không được — chỉ AWS S3) **hoặc** client-side combined (audio+screen từ Daily). Chốt khi tới.
3. **Phase 4 — Auth Supabase**: verify JWT ở Worker, gate routes + token, per-meeting authz.
4. **Sau nữa**: DO realtime, rate-limit, observability, R2 lifecycle, data residency, E2E key hardening.
5. Trước production: rotate `DAILY_API_KEY` + `wrangler secret put`, khoá CORS về origin thật, xử lý token hết hạn (họp > 4h).
