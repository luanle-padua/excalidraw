# 2026-06-05 — Chia sẻ màn hình (screen share) qua Daily.co — Phase 1

## Mục tiêu phiên

App đã có canvas chung, chat, phiên dịch, gọi audio — **còn thiếu chia sẻ màn hình** (1 người Present → mọi người xem), tính năng cơ bản của mọi nền tảng họp. Phiên này **bàn plan kỹ → chọn hạ tầng → code Phase 1**.

## Quyết định hạ tầng (bàn kỹ trước khi code)

Khảo sát toàn cảnh build-vs-buy cho media (SFU vs CPaaS):

- **Self-host SFU (LiveKit/mediasoup/Janus) → LOẠI:** chủ dự án là vibe coder, không team IT → không nuôi server media.
- **Cloudflare Realtime → LOẠI cho giờ:** rẻ + hợp hạ tầng CF, nhưng **không có recording sẵn** → phải tự xây "recorder bot" + 1 server 24/7 (vài tuần kỹ sư) → quá sức.
- **Zoom Video SDK → LOẠI:** mạnh + phủ TQ, nhưng web SDK đòi **SharedArrayBuffer / cross-origin isolation** → vỡ phần load R2/font/IFC cross-origin của app. Tích hợp khó nhất.
- **Agora → không cần:** mạnh nhất cho Trung Quốc, nhưng phiên chốt "quên China đi".
- **→ CHỌN Daily.co (managed):** WebRTC thuần (không vướng cross-origin isolation), recording sẵn, DX sạch, **zero-ops** (không nuôi server — chỉ thêm 1 hàm token vào Worker). Giá ~$0.0015–0.004/người/phút, free 10k phút/tháng.

Kiến trúc chốt: **Daily = ống dẫn media; socket Collab sẵn có = presence + single-share lock; lazy-join** (chỉ nối Daily khi có người share → tiết kiệm phút). Token mint server-side, `DAILY_API_KEY` chỉ Worker thấy (ở `worker/.dev.vars`, đã gitignore).

## QUYẾT ĐỊNH PHASE 3 — Recording (chốt 2026-06-05, bàn trước khi làm)

### Hiện trạng recording
`MeetingRecorder` (client-side) ghi **CHỈ audio mix**, nút **host-only** trong call controls; stop → **tải thẳng về máy host** (`.webm` opus, thư mục Downloads — `RecordingControls.tsx` `triggerDownload`). **KHÔNG** lưu server/R2, người khác không lấy được, không ghi màn hình/canvas.

### 2 ràng buộc người dùng nêu → định hình giải pháp
1. **Host là máy BẤT KỲ (có thể yếu)** → ghi client-side encode video trong trình duyệt = rủi ro lag/hỏng → **phải ghi SERVER-SIDE**.
2. **Dữ liệu họp BẢO MẬT, không phải ai cũng tải** → tải phải **qua cổng có auth + check quyền**, không link công khai.

### Quyết định: Daily cloud recording → copy R2 → tải qua Worker có auth
```
Daily QUAY (server-side, máy host bất kỳ vẫn OK)
  → webhook recording.ready-to-download
  → Worker copy file (Daily managed storage) → R2 PRIVATE + xoá bản Daily
  → Tải: CHỈ qua route Worker `GET /v1/recordings/:id`
       ↳ verify Supabase JWT + check membership cuộc họp → mới stream từ R2
  → KHÔNG phát link R2/Daily công khai
```
- **Daily KHÔNG ghi thẳng R2** (chỉ hỗ trợ AWS S3 qua IAM). → bắt buộc bước **copy** (tự động qua webhook, server-side). Dùng **storage managed của Daily** làm chỗ tạm → khỏi dựng AWS bucket riêng.
- **Vì sao lưu cuối ở R2 (không để S3/Daily):** **R2 egress = MIỄN PHÍ**, S3 egress ~$0.09/GB. Recording là video xem nhiều lần → egress là chi phí chính → R2 rẻ hơn hẳn (vd 200GB tải/tháng: S3 ~$18 vs R2 $0). Copy S3→R2 chỉ trả S3-egress 1 lần, sau đó xem free mãi.
- **Chi phí ghi Daily:** ~**$0.003/phút ≈ $0.18/giờ** (chỉ khi thực sự ghi). Đổi lấy: server-side tin cậy + máy host bất kỳ.

### Loại bỏ (a) client-side → R2
Rẻ nhất (~$0) nhưng **không chịu nổi 2 ràng buộc**: máy host yếu encode video không tin cậy; và vẫn cần auth-gate khi tải. → loại cho mục tiêu này.

### ⚠️ Hệ quả thứ tự — auth phải có trước
"Tải có kiểm soát" **bắt buộc Supabase auth**. → **Đổi thứ tự roadmap: làm Supabase Auth TRƯỚC, rồi Recording.**
- **Phase 3 (mới) = Supabase Auth** (verify JWT ở Worker, gate routes + token + tải recording, per-meeting membership).
- **Phase 4 (mới) = Recording** (Daily cloud → webhook → copy R2 → tải qua Worker auth + xem lại trong review-mode + ghi cả màn hình).
